# Plaky115 Surfaces

The toolkit ships three independent surfaces, each composed of three layers.

## Architecture

```
                     openapi/plaky115-dx.openapi.yaml
                                    │
                                    ▼
                    openapi/plaky115-operation-metadata.json
                                    │
       ┌────────────────────────────┼────────────────────────────┐
       ▼                            ▼                            ▼
   TS SDK                       Go CLI                       MCP Server
       │                            │                            │
   src/generated/types.ts      cli/internal/plakysdk/        mcp-server/src/runtime/
   (openapi-typescript)        operations.go (generated)      docs-index.ts (generated)
       │                            │                            │
   src/generated/operations/   cli/internal/cli/raw/         mcp-server/src/tools/raw/
   *.ts (one per op)           *.go (one per op)             *.ts (one per op)
       │                            │                            │
   src/client/* (hand-crafted) cli/internal/cli/dx.go        mcp-server/src/tools/curated/
                              (curated workflows)           (curated workflows)
                              cli/internal/plakydx/
                              (runtime helpers)
```

## Layers

1. **Generated types** — produced by `openapi-typescript` from the overlay. Pure type information, zero runtime. Regenerated by `npm run generate:types`.
2. **Generated low-level operations** — one TS module per operation (SDK), one Go method per operation (CLI), one MCP tool per operation. Produced by in-repo Node scripts that read `metadata.json` and emit deterministic, no-opinion wrappers around `fetch`/`net/http`. Regenerated by `npm run generate:operations`/`generate:cli`/`generate:mcp`.
3. **Hand-crafted curated surface** — the public API. Resource-oriented `PlakyClient` (TS SDK), curated cobra commands (Go CLI), curated MCP tools. Owns pagination iterators, retries, idempotency, errors, rate-limit budget, interceptors, webhook helpers, mutation planning, workflow composition.

## Surface status

Run `npm run status:surfaces` to print the current state of each surface. Pass `--strict` to treat any `legacy` surface as drift.

## Why three layers

- **Generated types** keep the SDK in sync with the API contract without hand-typed `Record<string, unknown>` everywhere.
- **Generated low-level** gives MCP raw tools and CLI raw commands a 1:1 mapping with operations, mechanically updated when the API evolves.
- **Hand-crafted curated** is what end users touch. Stable names, ergonomic flows, idiomatic per language.

Generated files always carry an `AUTO-GENERATED` header pointing to the regenerate command. Hand-crafted files live next to them, never inside them.
